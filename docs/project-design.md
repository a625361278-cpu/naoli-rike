# 脑力日课项目设计文档

## 1. 项目概述

**项目名称**：脑力日课

**项目类型**：H5 脑力训练小游戏合集

**技术方向**：TypeScript + Phaser

**首版目标**：做一个轻量、可每日重复游玩的脑力训练合集。玩家每天完成一组短训练，获得今日分数、能力维度反馈和历史趋势。

本项目以经典脑力训练类游戏的成熟玩法为核心参考，目标是在规则层面尽量还原经典训练体验，例如限时计算、颜色干扰、瞬间记忆、动态计数、连续心算和迷你逻辑题。项目不随意发明偏离经典体验的新玩法；创新主要放在界面、美术、包装、节奏和后续平台化体验上。

项目不使用原作素材，不复制原作 UI，不使用原作角色，不使用原作题库文本，不使用“博士监修”“川岛隆太”等授权背书或暗示。

项目核心原则：

- 经典玩法优先，规则设计要先对照成熟脑力训练项目，再做适合手机的调整。
- 玩法必须基于真实规则生成和真实判分。
- 不使用假数据伪造成绩。
- 不用默认值掩盖异常状态。
- 异常数据、异常状态优先暴露和记录，便于定位原因。
- 首版范围保持克制，优先把少量玩法做完整。

## 2. 产品定位与目标用户

### 2.1 产品定位

脑力日课是一款面向碎片时间的轻量训练小游戏。玩家不需要学习复杂规则，每次打开即可完成 1-5 分钟训练。

产品重点不是专业医学评估，而是休闲化的认知挑战、日常练习和自我记录。

### 2.2 目标用户

- 想每天做一点轻量脑力练习的普通用户。
- 喜欢短平快益智小游戏的玩家。
- 希望用手机快速完成休闲训练的用户。
- 后续可扩展到微信小游戏用户，但首版先以 H5 验证玩法。

### 2.3 成功标准

首版完成后应满足：

- 用户打开后 10 秒内理解今天要玩什么。
- 每个小游戏单局时长控制在 30-90 秒。
- 每局都有明确的题目、输入、判定、反馈和结算。
- 每日训练有完整闭环：开始、完成、评分、保存、查看历史。
- 代码结构支持后续迁移微信小游戏，而不是把逻辑写死在浏览器 API 中。

## 3. 首版玩法规划

首版包含 6 个小游戏。每个小游戏都以经典脑力训练玩法为原型，在手机 H5 上做输入和视觉适配。每个小游戏都必须具备独立规则、难度配置、计分逻辑和结算数据。

玩法复刻边界：

- 复刻核心规则、训练目标、节奏和判分思路。
- 不复制原作关卡题目、文本题库、画面布局、音效、美术资源和可识别 UI。
- 如果经典玩法依赖 DS 手写或麦克风，首版先改为触屏/键盘输入，保留训练目标。

### 3.1 极速计算

玩家在限时内回答基础加减乘除题。

训练目标：

- 计算速度
- 注意力
- 正确率控制

核心规则：

- 初始只出现一位数和简单二位数加减法。
- 难度提升后加入乘法、除法和更大数字。
- 每题只有一个正确答案。
- 错题计入错误数，不自动修正。

首版输入方式：

- 手机端使用数字键盘 UI。
- 桌面端支持键盘输入。

### 3.2 颜色冲突

屏幕显示一个颜色词，例如“红色”，但文字本身可能是蓝色、绿色或黄色。玩家需要选择文字真实显示的颜色，而不是文字含义。

训练目标：

- 反应速度
- 抑制干扰
- 专注力

核心规则：

- 每题展示一个颜色词和实际文字颜色。
- 玩家必须选择实际颜色。
- 正确后进入下一题，错误时给出明确反馈。

首版颜色集合：

- 红
- 蓝
- 绿
- 黄

### 3.3 瞬间记忆

多个数字或图形短暂出现，随后消失。玩家需要按指定规则回忆并点击。

训练目标：

- 短时记忆
- 视觉记忆
- 顺序处理

核心规则：

- 屏幕显示 3-7 个数字。
- 显示时间随难度变化。
- 数字隐藏后，玩家按从小到大顺序点击对应位置。
- 点击错误立即记录，不静默跳过。

### 3.4 进出计数

若干小人或图标进入、离开房间，玩家最后回答房间中剩余数量。

训练目标：

- 工作记忆
- 动态计数
- 注意力稳定性

核心规则：

- 初始房间数量为 0 或指定值。
- 每轮播放若干“进入”和“离开”事件。
- 事件结束后玩家输入最终数量。
- 生成题目时必须保证过程中数量不为负数。

### 3.5 连续心算

玩家看到一串连续运算，需要记住中间结果并给出最终答案。

训练目标：

- 工作记忆
- 连续计算
- 抗干扰能力

核心规则：

- 起始数字显示后，连续出现多个运算步骤。
- 例：8、+3、-2、x2。
- 玩家最后输入最终结果。
- 难度由步骤数量、数字范围和运算类型决定。

### 3.6 迷你数独 / 逻辑格

首版先做 4x4 迷你数独，不直接做完整 9x9。

训练目标：

- 逻辑推理
- 规划能力
- 持续专注

核心规则：

- 使用 4x4 网格。
- 每行、每列、每个 2x2 宫内数字不能重复。
- 首版使用可验证解的预制题库或生成器。
- 如果生成器无法保证唯一解，必须暴露错误，不允许输出不可验证题目。

## 4. 核心系统规划

### 4.1 每日训练

每日训练是主入口。系统每天推荐 3 个小游戏组成今日训练。

首版默认推荐组合：

- 极速计算
- 颜色冲突
- 瞬间记忆

推荐规则后续可扩展为根据玩家薄弱维度动态选择，但首版不做复杂推荐算法。

### 4.2 自由训练

玩家可以在训练列表中自由选择任意已解锁小游戏。

自由训练成绩计入单项最好成绩，但不影响当天每日训练完成状态，避免状态语义混乱。

### 4.3 评分系统

每局结算包含：

- 得分
- 正确数
- 错误数
- 正确率
- 用时
- 连击数
- 能力维度贡献

评分原则：

- 分数由真实表现计算。
- 不使用随机奖励伪造成绩。
- 错误和超时必须如实记录。
- 如果题目生成或判分状态异常，应中断并展示错误信息，而不是给出默认分数。

### 4.4 能力维度

首版使用 5 个能力维度：

- 计算
- 记忆
- 反应
- 专注
- 逻辑

每个小游戏映射到 1-3 个能力维度。每日训练完成后汇总今日能力表现。

### 4.5 历史记录

首版本地保存：

- 每日总分
- 每日完成状态
- 每个小游戏最近成绩
- 每个小游戏最好成绩
- 连续打卡天数

H5 阶段使用平台存储适配层，不在业务逻辑中直接调用 localStorage。微信小游戏迁移时替换为 wx storage。

### 4.6 解锁机制

首版默认开放：

- 极速计算
- 颜色冲突
- 瞬间记忆

后续通过每日训练完成天数解锁：

- 第 2 天：进出计数
- 第 3 天：连续心算
- 第 5 天：迷你数独

解锁状态必须来自真实存档，不允许用默认全开掩盖存档读取失败。

## 5. 技术路线

### 5.1 技术选型

首版采用：

- TypeScript
- Phaser
- Vite
- 本地 JSON 配置
- 平台适配层

选择 Phaser 的原因：

- 适合 2D H5 小游戏。
- 支持场景、输入、计时、资源加载和音频。
- 比纯 PixiJS 更接近完整游戏框架。
- 后续迁移微信小游戏时，核心 TypeScript 逻辑可保留。

### 5.2 代码结构

建议目录：

```text
src/
  core/
    rules/
    scoring/
    progress/
    daily/
  games/
    quick-math/
    color-conflict/
    instant-memory/
    flow-count/
    chain-calc/
    mini-sudoku/
  ui/
    home/
    training-list/
    result/
    history/
    settings/
  platform/
    adapter.ts
    h5-adapter.ts
    wechat-adapter.ts
  storage/
    storage-service.ts
    schemas.ts
  assets/
```

### 5.3 架构边界

`core` 只负责真实业务逻辑：

- 生成题目
- 校验答案
- 计算分数
- 更新训练进度
- 校验存档状态

`games` 负责 Phaser 场景：

- 展示题目
- 接收输入
- 播放反馈动画
- 调用 core 判分

`platform` 负责平台能力：

- 存储
- 音频
- 震动
- 分享
- 生命周期

业务逻辑不得直接依赖：

- window
- document
- localStorage
- wx
- Phaser 场景对象

## 6. UI 与美术方向

### 6.1 视觉定位

UI 采用“清爽训练仪表盘”风格。

关键词：

- 明亮
- 干净
- 轻量
- 有秩序
- 有一点学习感，但不医疗化

建议色彩：

- 主色：清爽蓝绿
- 辅色：暖黄色
- 危险/错误：柔和红
- 背景：浅灰白

### 6.2 页面结构

首版页面：

- 首页：今日训练、连续打卡、推荐训练卡片。
- 训练列表：6 个小游戏入口和能力标签。
- 游戏页：统一顶部计时和进度，中间玩法区域，底部输入区。
- 结算页：分数、正确率、用时、能力维度反馈。
- 历史页：最近 7 天曲线和单项最好成绩。
- 设置页：音效、震动、清除数据、版本信息。

### 6.3 图片资产策略

首版不使用原作素材。玩法规则可以对齐经典训练项目，但视觉表达必须原创。

后续如需图片，优先使用 gpt-image2 生成：

- 应用图标
- 启动页插画
- 训练入口插画
- 成就徽章
- 空状态插画

图像生成原则：

- 不生成类似 DS 原作界面。
- 不生成真实博士或类似川岛隆太形象。
- 不使用任天堂、DS、Brain Age 等可识别商标元素。
- 插画风格统一，适合移动端小尺寸展示。

## 7. H5 到微信小游戏迁移策略

首版虽然先做 H5，但必须从第一天保持可迁移结构。

### 7.1 可复用部分

迁移时应尽量保留：

- core 玩法逻辑
- 题目生成器
- 判分系统
- 能力维度系统
- 存档 schema
- 大部分 Phaser 场景逻辑

### 7.2 需要替换部分

迁移微信小游戏时重点替换：

- 平台启动入口
- 资源加载路径
- 存储实现
- 音频实现
- 分享能力
- 生命周期处理
- 包体与分包配置

### 7.3 迁移约束

实现阶段需要避免：

- 业务层直接读写 localStorage。
- 游戏逻辑依赖 DOM 节点。
- 用 CSS 动画承载核心玩法。
- 把 UI 状态写死在浏览器事件里。
- 使用浏览器专有 API 作为核心路径。

## 8. 分阶段开发路线

### 阶段一：项目基础与两项训练

目标：

- 初始化 TypeScript + Phaser + Vite 工程。
- 建立 core / games / ui / platform 分层。
- 完成首页、训练列表、结算页。
- 完成极速计算和颜色冲突。
- 完成本地存储适配层。

验收标准：

- H5 可运行。
- 两个小游戏可从首页进入并完成结算。
- 成绩真实保存。
- 刷新页面后历史记录仍可读取。

### 阶段二：补齐首版 6 项训练

目标：

- 完成瞬间记忆。
- 完成进出计数。
- 完成连续心算。
- 完成 4x4 迷你数独。
- 所有小游戏接入统一结算。

验收标准：

- 6 个训练入口可用。
- 每个小游戏有独立难度和评分。
- 异常题目生成会暴露错误，不产生假成绩。

### 阶段三：每日系统与历史趋势

目标：

- 每日推荐 3 项训练。
- 完成每日训练流程。
- 实现连续打卡。
- 实现最近 7 天历史曲线。
- 实现能力维度汇总。

验收标准：

- 每日训练完成状态准确。
- 自由训练不污染每日训练完成状态。
- 历史页能展示真实记录。

### 阶段四：UI 打磨与图片资产

目标：

- 统一视觉风格。
- 增加基础音效和动画反馈。
- 使用 gpt-image2 生成必要概念图。
- 优化移动端触控体验。

验收标准：

- 手机竖屏体验完整。
- 页面文字不溢出。
- 主要按钮和输入区域适合触屏。
- 图片资产风格统一且无版权风险。

### 阶段五：微信小游戏迁移准备

目标：

- 补全 wechat adapter。
- 检查 Phaser 微信小游戏兼容路线。
- 梳理资源包体和分包方案。
- 验证存储、音频、生命周期替换。

验收标准：

- core 不需要修改即可被微信入口调用。
- H5 adapter 和 wechat adapter 接口一致。
- 明确微信小游戏发布前剩余工作。

## 9. 风险与边界

### 9.1 版权与表达风险

项目要复刻经典脑力训练玩法的核心规则和训练体验，但不复制具体表现、素材、题库文本、商标元素或授权背书。

禁止：

- 使用原作名称。
- 使用原作 UI。
- 使用原作素材。
- 使用原作角色或博士形象。
- 使用原作题库文本。
- 暗示获得任天堂、东大、川岛隆太授权。

### 9.2 技术风险

主要风险：

- Phaser 到微信小游戏的适配成本。
- 移动端性能和包体。
- 本地存档损坏或版本迁移。
- 迷你数独生成器唯一解校验。

处理原则：

- 首版先做 H5 跑通真实玩法。
- 所有存档读写通过 adapter。
- 迷你数独可以先使用可验证题库，不强行上复杂生成器。
- 遇到异常状态优先报错和记录，不用兼容逻辑掩盖问题。

## 10. 首版范围总结

首版必须做：

- H5 可运行版本。
- 6 个小游戏。
- 每日训练。
- 自由训练。
- 统一结算。
- 本地历史记录。
- 连续打卡。
- 基础能力维度。

首版不做：

- 用户账号。
- 云同步。
- 排行榜。
- 付费系统。
- 广告系统。
- 语音识别。
- 手写识别。
- 完整 9x9 数独。
- 微信小游戏正式发布。

首版的目标是验证：用户是否愿意每天打开，完成 3 个短训练，并关心自己的历史变化。
