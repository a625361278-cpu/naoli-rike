# 脑力日课开发过程记录

本文用于保存早期未绑定到 `E:\naoli-rike` 项目线程里的关键讨论和实现过程。原始会话后来已重命名为“脑力日课项目开发记录”，但它最初是在 Codex 自动创建的临时目录下开始的，所以左侧项目过滤时不一定能直接看到。

## 1. 起点：参考对象与产品方向

最早讨论的参考对象是 NDS 上的《脳を鍛える大人のDSトレーニング》，也就是常说的《大人的 DS 脑力锻炼》/《Brain Age》。讨论后确认：

- 这个项目适合做成“每日脑力训练 + 小游戏合集”的轻量产品。
- 价值重点不是剧情或复杂画面，而是短训练、每日重复、成绩反馈和历史记录。
- 可以参考经典脑力训练的规则、节奏和训练目标，但不能复制原作名称、UI、美术、题库文本、博士形象、授权背书或可识别素材。

后续用户明确修正过边界：玩法规则应该尽量复刻经典训练体验，不能由 AI 随便发明；原创化主要发生在界面、美术、包装、命名和平台体验上。

## 2. 平台与技术选型

早期比较过几条路线：

- `uni-app` / `Taro`：适合跨端应用页面，但不是小游戏核心玩法的最佳方案。
- `Cocos Creator`：适合 H5 和微信小游戏正式发布，但用户不想为了当前项目再装一个 Cocos。
- `Godot`：能导 Web，但微信小游戏路线不是最稳的官方主线。
- `Unity`：用户熟悉，但普通 Unity WebGL 对手机 H5 不友好，且对本项目这种题目/计时/判分型产品偏重。
- `TypeScript + PixiJS/Phaser`：更轻，适合快速 H5 原型，也方便 AI 直接实现和维护。

最终选定：

- 当前首版使用 `TypeScript + Phaser + Vite`。
- 玩法核心尽量写成纯 TypeScript，避免直接绑死浏览器、DOM 或 Phaser。
- 平台能力如存储、音频、未来微信生命周期等通过适配层处理。
- H5 先验证玩法，后续再考虑微信小游戏迁移。

## 3. 项目命名与首版范围

项目名确定为“脑力日课”。选择原因：

- 不碰原作名称或授权暗示。
- 能表达“每天练一点”的产品循环。
- 适合 H5、微信小游戏和后续长期扩展。

首版决定只做 6 个训练，不一口气铺开所有 Brain Age 类玩法：

- 极速计算
- 颜色冲突
- 瞬间记忆
- 进出计数
- 连续心算
- 迷你数独 / 4x4 逻辑格

筛选原则：

- 适合触屏和键盘。
- 不依赖手写识别、语音识别或大量版权题库。
- 玩家容易理解，单局短，适合每日重复。
- 每个训练都必须有真实题目生成、真实判分和真实结算。

## 7. 答题抖动修复

用户反馈：选择答案后，题目都会抖动一下。

定位到两个原因：

- 错误反馈动画把 `transform: translateX(...)` 作用在整个 `.play-panel` 上，导致整块题目区域横向抖动。
- 新题目的 `.big-prompt` / `.color-word` 使用带位移和缩放的 `pop-in`，切换下一题时也会像抖动。

修复方式：

- 正确/错误反馈改成不改变位置的阴影高亮。
- 题目入场改成淡入，不再位移。

验证：

- `npm test` 通过：4 个文件，14 个测试。
- `npm run build` 通过。
- Playwright 验证答题后 `.play-panel` 的 `transform` 始终为 `none`。
- 截图输出到 `output/web-game/no-shake-after-answer.png`。

## 8. 颜色优先与限时紧张感改造

用户反馈：

- 颜色冲突比运算更有意思，应该放到第一位。
- 当前答题没有紧张感，玩家不会感到要快点答、慢了会没时间或分数低。

真实实现检查后确认：

- 当时每日训练和训练列表都是极速计算排第一。
- 计分内部虽然有速度加成，但 UI 没有倒计时/剩余时间展示，玩家感受不到压力。

改造目标：

- 颜色冲突成为第一主推玩法。
- 所有训练加入明确限时。
- 游戏页显示剩余秒数和时间条。
- 超时自动按当前真实表现结算。
- 评分强化剩余时间奖励。

主要改动：

- `getDailyTrainingIds()` 改为颜色冲突优先。
- `allTrainingIds` 和训练列表展示顺序改为颜色冲突第一。
- 每个训练增加 `timeLimitMs`：
  - 颜色冲突：30s
  - 极速计算：45s
  - 瞬间记忆：60s
  - 进出计数：50s
  - 连续心算：55s
  - 迷你数独：120s
- `SessionState` 增加 `timeLimitMs`、`deadlineAt`、`timeLeftMs`。
- `buildResultFromSession()` 支持 `timeLimitMs` 并强化剩余时间分。
- 游戏页增加倒计时 chip 和 timer bar。
- 低于 10 秒时用非位移动画提示，避免题目区域移动。
- 超时自动结算，不补默认成绩，不伪造完成率。
- 计时器改为只更新秒数和时间条 DOM，不每 200ms 全量重绘题目区，避免按钮被频繁替换导致点击不稳。
- 修复 0 答题超时时能力维度仍有最小值的问题，避免伪造能力表现。

验证结果：

- `npm test` 通过：4 个文件，18 个测试。
- `npm run build` 通过。
- Playwright 确认“开始今日训练”进入颜色冲突。
- Playwright 确认游戏页显示剩余秒数和时间条。
- Playwright 确认快速颜色冲突得分高于慢速。
- Playwright 确认超时自动 0 分结算。
- Playwright 跑完 6 个限时训练并进入结算。
- 检查移动端颜色训练截图。

## 10. 重要原则

后续继续开发时应保持这些原则：

- 不用假数据伪造成绩。
- 不用默认值掩盖异常状态。
- 题目生成、判分、计分、存档都必须暴露真实问题。
- 经典玩法规则优先，UI 和美术原创化。
- 自由训练不能污染每日训练完成状态。
- 超时、错误、未答题都必须如实进入结果语义。
- 核心玩法逻辑尽量保持在 `src/core/*`，平台和渲染层不要反向污染业务规则。
